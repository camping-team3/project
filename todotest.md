# 🧪 디자인 시스템 검증 가이드 (Design System Testing)

본 문서는 설계된 디자인 토큰과 컴포넌트 룰이 프로젝트의 표준으로서 적합한지, 그리고 실제 구현에 빈틈이 없는지 확인하기 위한 테스트 방법론을 정의합니다.

---

## 1단계: 설계 품질 테스트 (Logic & Strategy Verification)
코드를 짜기 전, 우리가 만든 '약속' 자체가 튼튼한지 검증합니다.

### ✅ 시나리오 스트레스 테스트 (The "What-If" Test)
- **방법**: "만약 브랜드 컬러가 초록색에서 오렌지색으로 바뀐다면?" 가정하기.
- **통과 기준**: `--fh-primary` 값 하나만 수정했을 때, 서비스 전체의 주요 포인트가 어색함 없이 오렌지로 변해야 함.
- **체크**: 변수명이 색상 이름(예: `--green`)이 아닌 역할 이름(예: `--primary`)으로 되어 있는가?

### ✅ 블라인드 구현 테스트 (The "Blind Dev" Test)
- **방법**: 새로운 팀원에게 `todo2.md`, `todo3.md`만 주고 "새로운 페이지"를 그려보게 함.
- **통과 기준**: 추가 질문 없이도 기존 페이지와 90% 이상의 일관성을 가진 UI가 결과물로 나와야 함.
- **체크**: "여백은 몇 px인가요?", "폰트 크기는요?" 같은 질문이 나오면 룰이 너무 추상적인 것임.

### ✅ 확장성 테스트 (Scalability Test)
- **방법**: "나중에 다크 모드를 도입한다면?" 가정하기.
- **통과 기준**: 색상 토큰이 배경색(`--fh-bg-light`), 텍스트색(`--fh-text-main`)처럼 역할 중심으로 정의되어 있어 값만 바꾸면 대응이 가능해야 함.

---

## 2단계: 구현 및 적용 테스트 (Implementation Verification)
만들어진 룰이 실제 코드에 얼마나 잘 녹아들었는지 검증합니다.

### ✅ 시각적 카탈로그 테스트 (Style Guide Page)
- **방법**: 프로젝트 내에 모든 컴포넌트(버튼, 카드, 테이블, 폰트)를 한데 모은 '스타일 가이드' 페이지를 생성.
- **통과 기준**: 한 화면에서 모아봤을 때 톤앤매너가 일치하고, CSS 변수가 누락되어 색상이 깨지는 곳이 없어야 함.

### ✅ 하드코딩 전수 조사 (Grep Search Test)
- **방법**: 전체 CSS 파일에서 `#` (Hex code) 또는 `px` 단위가 직접 사용된 곳을 검색.
- **통과 기준**: `variables.css`를 제외한 파일에서 하드코딩된 스타일 값이 최소화되어야 함 (모두 `var(--...)`로 대체 권장).

### ✅ 커버리지 갭 분석 (Coverage Gap Analysis)
- **방법**: 기존 공지사항/갤러리 페이지를 띄워놓고 모든 요소를 우리가 만든 토큰으로 설명할 수 있는지 대조.
- **통과 기준**: "이 간격은 우리 토큰에 없는 13px인데?" 같은 예외 상황이 없어야 함.

---

## 3단계: AI/동료 리뷰 (Review Process)
- **방법**: 작업한 UI 코드를 AI나 동료에게 보여주며 `design-rule.md` 준수 여부 검토 요청.
- **통과 기준**: 리뷰어로부터 "기존 디자인 시스템의 규칙을 완벽히 따르고 있다"는 확답을 얻어야 함.
