## 🏕️ 캠핑장 예약 시스템 아키텍처 및 설계 명세

### 1. 시스템 인프라 구조 (System Infrastructure)

- **단일 애플리케이션 서버:** 초기 트래픽과 유지보수 비용을 고려하여 단일 Spring Boot 인스턴스에서 구동합니다.
- **인메모리 데이터 저장소 (Redis):** 서버 재시작 시에도 로그인 세션을 안전하게 유지하고, 예약 선점 락(Lock)을 DB 부하 없이 빠르고 가볍게 처리하기 위해 도입합니다.
- **관계형 데이터베이스 (MySQL):** 회원, 예약 내역, 결제, 게시판 데이터 등 영구적으로 보존되고 정합성이 보장되어야 하는 핵심 비즈니스 데이터를 관리합니다.

### 2. 동시성 제어 및 상태 관리 (Concurrency & State Management)

- **Redis TTL 기반 임시 선점 (Lock):** 회원이 특정 객실의 [결제하기] 화면에 진입하면, 해당 일정과 객실 ID를 조합한 키를 Redis에 생성하고 5분의 만료 시간(TTL)을 부여합니다. 시간이 지나면 Redis에서 자동으로 데이터가 증발하므로 별도의 해제 로직이나 MySQL의 부하가 없습니다.
- **지연 평가 (LAZY Evaluation) 상태 업데이트:** 백그라운드 스케줄러가 계속 돌면서 DB를 수정하지 않습니다. 고객이나 관리자가 페이지를 '조회하는 시점'에 오늘 날짜와 예약 일자를 비교하여, 조건에 맞으면 화면에 '이용 완료'로 렌더링하고 리뷰 작성 버튼을 활성화하는 방식을 사용해 서버 자원을 효율적으로 아낍니다.

### 3. 스토리지 및 파일 관리 (Storage)

- **로컬 파일 시스템 (Local Storage):** 리뷰 사진, 포토 갤러리 이미지 등은 외부 스토리지 서비스(S3 등)를 사용하지 않고 애플리케이션이 구동되는 서버의 로컬 디렉토리에 직접 저장합니다. MySQL 테이블에는 저장된 파일의 물리적/상대적 경로(URL) 텍스트만 기록합니다.

### 4. 보안 및 로깅 체계 (Security & Logging)

- **인증 및 인가 (HttpSession + Interceptor):** 로그인 성공 시 유저 객체를 세션에 담습니다. Spring의 Interceptor를 구성하여 `/admin/**` 경로는 관리자 권한을 가진 세션만, `/mypage/**` 및 `/reservation/**` 경로는 로그인된 세션만 접근하도록 URL 라우팅 단계에서 철저히 차단합니다.
- **파일 기반 로깅 (File-based Logging):** 결제 실패, 시스템 예외, 오류 내역 등은 Logback 등을 통해 서버 내의 텍스트 파일(`.log`)로만 저장합니다. 관리자 화면에는 에러 로그를 노출하지 않으며, 필요시 엔지니어가 직접 서버에 접속해 확인합니다.

### 5. 통신 패턴 (Communication Pattern)

- **SSR 기반 화면 전환:** Mustache를 사용해 컨트롤러에서 최종 완성된 HTML을 내려줍니다.
- **Form Submit 및 PRG 패턴:** 데이터의 생성/수정/삭제 요청은 모두 `<form method="POST">`로 전송하며, 처리가 끝나면 반드시 다른 페이지로 리다이렉트(GET) 시켜 새로고침 시 데이터가 중복 전송되는 것을 막습니다.
- **제한적 비동기 통신 (AJAX):** 예약 화면의 실시간 달력 객실 조회, 회원가입 시 이메일 중복 체크 등 폼 제출로 해결할 수 없는 극히 일부 기능에만 사용합니다.
