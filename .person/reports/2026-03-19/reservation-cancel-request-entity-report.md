# Phase 3 Reservation 도메인 확장 (1단계: ReservationCancelRequest 엔티티 생성)

**Date**: 2026-03-19
**Reporter**: parkcoding
**Task**: 1-3. ReservationCancelRequest 엔티티 생성

## 1. 작업 요약
고객의 예약 취소 요청 사유와 환불 관련 정보를 관리자 승인 전까지 안전하게 보관하기 위한 전용 엔티티를 생성했습니다.

## 2. 변경 사항
- **환불 데이터 중심 설계**: 단순히 취소 사유뿐만 아니라 환불 금액(`refundAmount`), 환불 계좌 정보(`refundBank`, `refundAccount`, `refundAccountHolder`) 필드를 포함하여 추후 환불 로직 연동을 위한 기반을 마련했습니다.
- **도메인 일관성 확보**: `ReservationChangeRequest` 엔티티와 동일한 `RequestStatus` 체계 및 `approve()`, `reject()` 캡슐화 로직을 적용하여 전체적인 설계 통일성을 갖추었습니다.
- **데이터 저장 최적화**: 취소 사유(`reason`)를 최대 1000자까지 저장할 수 있도록 길이를 지정하여 긴 요청 메시지도 손실 없이 수용 가능하게 했습니다.
- **공통 기반 상속**: `BaseTimeEntity`를 상속받아 요청 생성일과 수정일이 자동으로 기록되도록 했습니다.

## 3. 검증 결과
- `Reservation` 엔티티와의 연관관계가 지연 로딩(`LAZY`)으로 올바르게 설정되었습니다.
- 환불 관련 핵심 필드들이 적절한 타입(금액은 Long 등)으로 정의되었습니다.

## 4. 비유로 보는 작업 내용
기존 예약서 위에 '취소'라고 낙서하는 것이 아니라, **'환불 및 예약 취소 신청서'**를 별도로 만든 것과 같습니다. 이 신청서에는 "왜 취소하고 싶은지" 뿐만 아니라, "어느 계좌로 얼마를 돌려받아야 하는지" 적을 수 있는 전용 칸을 마련했습니다. 관리자는 이 서류를 꼼꼼히 검토한 뒤 승인 도장을 찍을 수 있게 된 셈입니다.
