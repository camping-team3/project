# Phase 3 Reservation 도메인 확장 (1단계: Reservation 엔티티 연관관계 추가)

**Date**: 2026-03-19
**Reporter**: parkcoding
**Task**: 1-4. Reservation 엔티티 연관관계 추가

## 1. 작업 요약
`Reservation` 엔티티에서 예약 변경 및 취소 요청 이력을 직접 조회할 수 있도록 양방향 연관관계를 설정하고, 관련 비즈니스 메서드를 추가했습니다.

## 2. 변경 사항
- **양방향 연관관계 설정**: `changeRequests`, `cancelRequests` 필드를 추가하고, 상세한 주석과 함께 `mappedBy`, `cascade`, `orphanRemoval` 설정을 적용하여 데이터 정합성을 확보했습니다.
- **기존 코드 보존**: 수동으로 작성된 `@Builder` 생성자를 그대로 유지하여 기존 시스템과의 호환성을 100% 보장했습니다.
- **비즈니스 메서드 추가**:
    - `updateStatus()`: 예약 상태를 안전하게 변경하기 위한 메서드
    - `updateReservationInfo()`: 관리자 승인 시 새로운 예약 정보(날짜, 사이트, 인원 등)를 한꺼번에 업데이트하기 위한 메서드
- **데이터 초기화**: 연관관계 필드 선언 시 `ArrayList`로 초기화하여 NPE를 방지했습니다.

## 3. 검증 결과
- 양방향 매핑 설정이 JPA 컨벤션에 따라 올바르게 적용되었습니다.
- 기존 생성자의 필드 매핑 로직이 손상되지 않았음을 확인했습니다.

## 4. 비유로 보는 작업 내용
기존에는 예약서(Reservation)와 변경 신청서(Request)가 각각 따로 노는 서류였다면, 이제는 **예약서 뒷면에 '관련 신청서 목록'이라는 부록 칸**을 만든 것과 같습니다. 이 부록 칸을 통해 이 예약과 관련된 모든 변경/취소 기록을 한눈에 찾아볼 수 있게 되었고, 부모 서류(예약서)가 파기되면 부록들도 함께 정리되도록 규칙을 정한 셈입니다.
