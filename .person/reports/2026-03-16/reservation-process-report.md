## 📝 작업 요약
사용자가 선택한 사이트를 기반으로 결제 상세 정보를 제공하고, 실제 예약 데이터를 서버 측 검증을 거쳐 안전하게 저장하는 프로세스를 구현했습니다. 특히 인원 추가 요금 정책(인당 10,000원)을 서버 로직에 엄격히 적용했습니다.

## 🛠️ 주요 변경 사항
### 1. 결제 데이터 구조화 (DTO & Controller)
- **PaymentFormDTO**: 결제 페이지 렌더링에 필요한 사이트 정보, 박수, 인원별 추가 요금, 최종 합계를 구조화하여 전달합니다.
- **PRG 패턴 적용**: 예약 성공 후 결제 완료 페이지로 리다이렉트하여 중복 제출을 방지했습니다.

### 2. 철저한 서버 측 검증 (Service)
- **가격 재계산**: 클라이언트에서 넘어온 금액을 믿지 않고, 서버에서 `(기본가 + 초과인원*1만)*박수` 공식으로 다시 계산하여 데이터 무결성을 보장했습니다.
- **최종 중복 체크**: 사용자가 결제 버튼을 누르는 찰나에 다른 사람이 먼저 예약했는지 `existsBySiteIdAndDateRange` 쿼리로 한 번 더 확인합니다.

### 3. 결제 뷰(View) 동적 연동
- `payment.mustache`에 실제 예약 데이터를 바인딩하고, 숨겨진 필드(Hidden input)를 통해 예약에 필요한 모든 정보를 서버로 안전하게 POST 전송하도록 구성했습니다.

## 💡 비유로 설명하기
**"이중 잠금 장치가 있는 계약서 작성"**
손님이 카운터에서 방을 고르고 결제하러 가는 동안, 저희 시스템은 뒤에서 두 가지 일을 합니다. 첫째, 손님이 계산한 금액이 맞는지 주판을 다시 튕겨봅니다(가격 재계산). 둘째, 손님이 사인하기 직전에 그새 다른 사람이 그 방 키를 가져갔는지 장부를 다시 확인합니다(중복 체크). 이 모든 확인이 끝나야 비로소 '예약 대기' 도장을 쾅 찍어 장부에 기록하는 것과 같습니다.
