# 🏕️ 캠핑 구역 및 사이트(Site) 도메인 종합 구현 보고서

## 1. 개요
캠핑장 예약 시스템의 핵심인 '구역(Zone)'과 '사이트(Site)' 도메인의 사용자 화면 연동부터 관리자 CRUD 기능, 그리고 안정적인 데이터 처리를 위한 인프라 설정까지의 전체 구현 과정을 정리한 종합 보고서입니다.

## 2. 주요 구현 내용

### 2.1 사용자 인터페이스 (User-Facing)
- **메인 페이지 (`index`)**: `SiteRepository`에서 `Fetch Join`을 통해 구역 정보를 한 번에 가져와 캠핑 사이트 목록을 동적으로 렌더링하도록 구현했습니다. (N+1 문제 해결)
- **상세 페이지 (`site/detail`)**: 각 사이트의 상세 정보(수용 인원, 요금 등)를 ID 기반으로 조회하여 연동했습니다.
- **예약 필터링 (`reservation/new`)**: 사용자가 선택한 날짜에 이미 예약된 건(`PENDING`, `CONFIRMED`, `CANCEL_REQ`)을 제외하고 가용 사이트만 보여주는 `NOT IN` 기반의 JPQL 쿼리 로직을 완성했습니다.

### 2.2 관리자 기능 (Admin CRUD)
- **통합 관리 환경**: `AdminController`를 통해 구역과 사이트의 등록, 수정, 삭제(6종 핸들러)를 처리하며, 부트스트랩 모달을 활용해 페이지 이동 없는 실시간 관리 UI를 구축했습니다.
- **안전한 데이터 변경**: 모든 POST 작업 후 PRG 패턴을 적용하여 새로고침 시 중복 실행을 방지하고, 엔티티 내 `update()` 메서드를 통한 캡슐화된 정보 수정을 지향했습니다.

### 2.3 데이터베이스 및 인프라 최적화
- **명명 전략 수정**: 스네이크 케이스(`created_at`) 인식을 위해 `CamelCaseToUnderscoresNamingStrategy`를 적용했습니다.
- **초기화 순서 제어**: H2 DB 환경에서 테이블 생성 후 `data.sql`이 안정적으로 실행되도록 `defer-datasource-initialization: true` 설정을 보강했습니다.
- **참조 무결성 (Cascade)**: `Zone` 삭제 시 하위 `Site`들이 자동으로 연쇄 삭제되도록 `CascadeType.ALL` 및 `orphanRemoval = true`를 설정하여 데이터 일관성을 확보했습니다.

## 3. 검증 결과
- **정합성**: 관리자에서 수정한 정보(요금, 이름 등)가 메인, 상세, 예약 화면에 즉시 반영됨을 확인했습니다.
- **필터링**: 예약 내역이 있는 기간 검색 시 해당 사이트가 목록에서 정확히 제외되는 시나리오를 검증했습니다.
- **삭제 정책**: 사이트가 포함된 구역 삭제 시 외래 키 제약 조건 에러 없이 정상적으로 연쇄 삭제됨을 확인했습니다.

## 4. 비유로 보는 종합 요약
이전의 시스템이 **"사진만 붙어있는 고정된 메뉴판과 수동으로 확인해야 하는 종이 명부"**였다면, 이번 작업을 통해 캠핑장 주인은 **"언제든 구역을 늘리고 요금을 조정할 수 있는 전용 매직 태블릿"**을 갖게 되었고, 고객은 **"빈 자리를 실시간으로 찾아 예약할 수 있는 디지털 키오스크"**를 이용할 수 있게 된 것과 같습니다.
