## 1. 작업 요약 (Executive Summary)

본 작업은 `dev` 브랜치 머지 과정에서 발생한 **컨플릭트 잔재**와 **구조적 중복**을 제거하고, 결제 프로세스에서 발생하던 **치명적인 세션 타입 불일치 버그**를 해결하여 시스템의 기초 안정성을 확보하는 것을 목적으로 수행되었습니다.

## 2. 주요 변경 사항 (Key Changes)

### 2.1 컨플릭트 마커 제거 및 템플릿 로직 복구
- **대상**: `reservation/new.mustache`, `reservation/payment.mustache`, `admin/gallery/update-form.mustache`
- **내용**: 
    - 소스 코드 내부에 남아있던 `<<<<<<<`, `=======`, `>>>>>>>` 마커를 모두 제거했습니다.
    - **방문객 정보 입력 필드**와 **인원 추가 요금 동적 계산** 로직을 HEAD 브랜치 기준으로 통합하여 기능적 완성도를 회복했습니다.

### 2.2 중복 프로젝트 구조(`project/` 폴더) 정리
- **내용**: 머지 실수로 루트 디렉토리에 생성된 구 버전 프로젝트 폴더(`project/`)를 전체 삭제했습니다. 이를 통해 개발 환경의 혼란을 방지하고 단일 프로젝트 구조로 정상화했습니다.

### 2.3 결제 시 세션 불일치 버그 수정
- **문제**: 로그인 시 세션에는 `UserResponse.LoginDTO`가 저장되는데, `ReservationController`에서는 이를 `User` (엔티티) 타입으로 꺼내려다 `ClassCastException`이 발생했습니다.
- **수정**: 
    - `ReservationController.reserve()`에서 세션 유저 타입을 `LoginDTO`로 변경했습니다.
    - `ReservationService.reserve()`에서 `LoginDTO`의 ID를 기반으로 실제 `User` 엔티티를 다시 조회하여 연관관계를 안전하게 맺도록 수정했습니다.

### 2.4 통합 테스트 코드 동기화
- **대상**: `ReservationIntegrationTest.java`
- **내용**: 변경된 서비스 규격(`LoginDTO` 파라미터)에 맞춰 테스트 코드를 수정하고, `LoginDTO` 생성 시 빌더 오류를 생성자 호출 방식으로 교정했습니다.

## 3. 검증 결과 (Verification Results)

- **컨플릭트 검색**: 프로젝트 전체에서 머지 마커 재검색 결과, 유해한 잔재가 0건임을 확인했습니다.
- **구조 무결성**: `project/` 폴더 삭제 후 루트 디렉토리 기반의 빌드 환경이 정상임을 확인했습니다.
- **로직 검증**: 세션 캐스팅 오류가 해결되어, 로그인 상태에서 결제하기 클릭 시 로그인 페이지로 튕기지 않고 결제 완료 프로세스가 정상 작동함을 논리적으로 검증했습니다.

## 4. 비유를 통한 기술 설명 (Technical Metaphor)

**"잘못된 열쇠로 문 열기"**
기존의 버그는 마치 **'도어락(세션)'**에 등록된 정보는 **'지문(LoginDTO)'**인데, 정작 문을 열 때 시스템이 **'열쇠(User 엔티티)'**를 내놓으라고 강요하는 것과 같았습니다. 

지문(DTO)을 대도 시스템이 열쇠(엔티티)가 아니라고 에러를 내며 사용자를 쫓아냈던 상황(로그인 페이지로 리다이렉트)을, 이제는 **지문 인식기(DTO 처리 로직)**를 제대로 설치하고, 지문이 확인되면 **내부 금고에서 열쇠(엔티티 조회)**를 꺼내 문을 열어주는 방식으로 고친 것입니다.
