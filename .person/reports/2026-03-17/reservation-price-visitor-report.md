# Reservation 가격 계산 고도화 및 방문객 정보 수집 리포트

**Date:** 2026-03-17  
**Reporter:** parkcoding  
**Status:** Phase 2 Task 7 완료

---

## 📝 작업 요약
예약 시스템의 가격 계산 로직을 하드코딩에서 `Zone` 엔티티 기반의 동적 데이터로 전환하고, 실제 캠핑장 이용 시 필수적인 **방문객 정보(성함, 연락처)** 수집 기능을 구현했습니다.

## 🛠️ 주요 변경 사항

### 1. 데이터 구조 및 DTO 개선 (Refactoring)
- **`SiteResponse.ResevationAvailableListDTO`**: `Site` 엔티티를 직접 전달받아 초기화하도록 생성자를 리팩토링했습니다. 이제 구역별 기준 인원(`basePeople`)과 추가 요금(`extraPersonFee`)을 정확히 포함합니다.
- **`ReservationRequest.ReserveDTO`**: 사용자가 입력한 `visitorName`과 `visitorPhone`을 전달받기 위한 필드를 추가했습니다.
- **`ReservationResponse.PaymentFormDTO`**: 결제 화면에서 동적인 추가 요금 안내를 위해 `extraPersonFee` 필드를 추가하고, 모든 숫자 타입을 `Long`/`Integer` 래퍼 타입으로 통일했습니다.

### 2. 비즈니스 로직 고도화 (Service)
- **가격 계산 동적화**: 기존에 10,000원으로 고정되어 있던 추가 인원 요금을 각 `Zone`에 설정된 `extraPersonFee`를 사용하도록 수정했습니다.
- **방문객 정보 저장**: 예약 생성 시(`reserve`) 전달받은 방문객 정보를 `Reservation` 엔티티에 매핑하여 DB에 영구 저장합니다.
- **람다식 적용**: `findAvailableSites`에서 스트림 처리를 메서드 참조 방식에서 람다식(`site -> new SiteResponse.ResevationAvailableListDTO(site)`)으로 명시적으로 수정했습니다.

### 3. 사용자 인터페이스 연동 (View & JS)
- **실시간 가격 계산 (JS)**: `reservation/new.mustache`에서 사이트 선택 시, 하드코딩된 값 대신 서버에서 내려준 해당 사이트의 추가 요금 데이터를 기반으로 총액을 계산합니다.
- **정보 입력 칸 추가**: `reservation/payment.mustache`에 방문객 정보를 입력받는 섹션을 추가했습니다. 기존 디자인 레이아웃을 100% 보존하면서도 깔끔한 입력 폼을 배치했습니다.

## ✅ 검증 결과
- **구역별 요금 차등 적용**: 추가 요금이 5,000원인 구역과 10,000원인 구역이 각각 선택된 인원에 따라 정확히 계산됨을 확인했습니다.
- **방문객 정보 전달**: 결제 전 정보를 입력하지 않으면 알럿 창이 뜨며, 입력 시 DB의 `visitor_name`, `visitor_phone` 컬럼에 정확히 저장됨을 확인했습니다.

## 💡 어려운 내용 설명 (비유로 이해하기)

### "뷔페 가격표 교체하기"
이전에는 캠핑장 직원이 "추가 인원은 무조건 만 원입니다"라고 적힌 종이만 보고 계산했다면(하드코딩), 이제는 **구역마다 다른 메뉴판(Zone 정보)**을 보고 계산기로 정확히 두드려주는 시스템으로 바뀐 것입니다.

### "택배 송장 붙이기"
예약만 달랑 받는 것이 아니라, 실제로 누가 올지 적어두는 **송장(방문객 정보)**을 예약증에 꼭 붙여서 저장하도록 만들었습니다. 이제 관리자는 "누가 올지" 미리 알 수 있게 되었습니다.
