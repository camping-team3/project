# [종합 보고서] 독립형 블랙리스트 시스템 및 맞춤형 차단 사유 안내 기능 구현

**작성자:** Gemini-CLI  
**날짜:** 2026-03-24  
**주제:** 코어 코드 보존 기반의 도메인 보안 확장 및 동적 피드백 시스템 구축

---

## 1. 구현 기능 요약 (What we built)

### ① 블랙리스트 전용 영구 저장소 (`blacklist_tb`)
- **개념**: 유저가 `ANONYMOUS`(마스킹) 상태가 되어도, 차단된 아이디와 구체적인 사유를 별도로 관리하는 독립 테이블을 신설했습니다.
- **데이터 보존**: 유저 엔티티와의 물리적 관계를 끊음으로써 유저 데이터가 삭제되거나 변형되어도 블랙리스트 기록은 안전하게 유지됩니다.

### ② 관리자 맞춤형 차단 사유 수집 (Dynamic Input)
- **UI 확장**: 관리자가 [강제탈퇴] 클릭 시, 텍스트 입력창(Modal)을 통해 직접 차단 사유를 입력할 수 있게 개선했습니다.
- **실시간 연동**: 관리자가 입력한 문구 그대로가 서버를 거쳐 DB에 영속화됩니다.

### ③ 지능형 로그인 차단 가이드 (Customized Alert)
- **선제적 방어**: 로그인 프로세스 최상단에 블랙리스트 체크 로직을 배치하여 불필요한 인증 과정을 생략하고 보안성을 높였습니다.
- **맞춤형 메시지**: 단순히 "로그인 불가"가 아니라, **"귀하는 [관리자 입력 사유]로 인해 이용이 제한되었습니다"**와 같이 사용자가 납득할 수 있는 구체적인 피드백을 제공합니다.

## 2. 핵심 기술 및 용어 설명 (Glossary)

| 용어 | 설명 |
| :--- | :--- |
| **Domain Isolation (도메인 격리)** | 기존의 `UserStatus`나 `Enum` 등 코어(Global) 코드를 수정하지 않고 새로운 기능(Blacklist)을 도메인 단위에서 덧붙이는 고도화된 설계 기법입니다. |
| **User Experience (UX) Feedback** | 시스템 중심의 딱딱한 메시지 대신, 관리자의 실제 의도(사유)를 전달하여 사용자 경험을 개선하는 방식입니다. |
| **Exception Translation** | 백엔드에서 발생한 `Exception400`과 메시지를 프론트엔드(`alert`)로 변환하여 사용자에게 전달하는 흐름을 의미합니다. |

---

## 3. 구현 과정 요약 (Implementation Highlights)

### [Back-end] `UserService.java`
- `blacklistRepository.findByUsername()` 호출을 통해 로그인 전 차단 여부 판별.
- `expelUser()` 메서드 확장으로 `Blacklist` 엔티티 동시 저장 구현.

### [Front-end] `admin/review/list.mustache`
- `SweetAlert2`의 `input: 'textarea'` 옵션을 사용하여 관리자의 입력 편의성 증대.
- `form.submit()`을 통한 비동기 데이터 전달 로직 최적화.

## 4. 조언 및 향후 확장성

이번 작업의 가장 큰 가치는 **"코어를 지키면서 기능을 확장했다"**는 점에 있습니다. 

향후 이 시스템은 다음과 같이 더 발전할 수 있습니다:
- **차단 기간 설정**: `blacklist_tb`에 `expiredAt` 컬럼을 추가하여 기간제 차단 기능 구현.
- **차단 해제 관리**: 관리자 페이지에 블랙리스트 목록 뷰를 만들어 선택적 차단 해제 기능 추가.

이제 사용자님의 시스템은 훨씬 더 정교하고 친절한(하지만 단호한) 보안 체계를 갖추게 되었습니다. 고생 많으셨습니다! 😌
