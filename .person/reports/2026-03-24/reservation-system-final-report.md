## 1. 개요
고객의 예약 변경/취소 과정에서 발생하는 정산(추가 결제 및 부분 환불) 시스템의 결함을 해결하고, 실시간 예약 시스템의 가용성 판단 로직을 정교화하였습니다. 또한 보안 설정(CSRF)과 UI/UX 간의 불일치를 해소하여 안정적인 운영 환경을 구축했습니다.

## 2. 주요 개선 및 해결 사항

### 2.1 사이트 가용성 판단 로직 정교화
- **문제**: 예약 변경 시 '동일 사이트 내 기간 변경'을 시도할 경우, 자신의 기존 예약 점유를 중복으로 판단하여 변경이 불가능했던 현상.
- **해결**:
  - `ReservationRepository`에 `currentReservationId`를 제외하고 가용성을 체크하는 쿼리 추가.
  - 프론트엔드(`reservation-change.mustache`)에서 조회 시 현재 예약 ID를 API에 함께 전달하도록 보완.

### 2.2 정산 시스템의 재무적 정합성 확보 (Settlement Integrity)
- **문제**: 관리자가 예약을 승인하는 즉시 예약 총액이 업데이트되어, 이후 고객이 환불을 요청할 때 차액 계산 기준이 소실되는 논리적 모순 발생.
- **해결**:
  - `ReservationChangeRequest` 엔티티에 `oldTotalPrice`(변경 전 금액) 필드를 추가하여 정산 기준점을 보존.
  - 관리자 승인 여부와 상관없이 신청 시점의 가격을 기준으로 정확한 환불액 산출 가능하도록 `RefundService` 로직 전면 수정.

### 2.3 고객 주도 환불 워크플로우 구축
- **문제**: 관리자 승인 후 자동 환불 시 API 오류 대응이 어렵고 고객에게 명확한 피드백 전달이 부족함.
- **해결**:
  - **2단계 프로세스 도입**: [관리자 승인] -> [고객이 상세페이지에서 '환불받기' 클릭] -> [실제 API 호출 및 상태 확정].
  - `isRefunded` 플래그를 통해 중복 환불을 방지하고 환불 완료 상태를 영속적으로 관리.

### 2.4 보안 정책 및 데이터베이스 동기화
- **CSRF 정책 정렬**: `SecurityConfig`에서 CSRF가 비활성화된 환경에 맞춰 JavaScript의 불필요한 토큰 검증 로직 제거 및 403 에러 방지.
- **DB 초기화 오류 수정**: 엔티티 확장에 따른 `data.sql`의 컬럼 누락(`is_refunded`, `old_total_price` 등) 및 PK 중복(ID 21) 문제를 해결하여 애플리케이션 기동 안정성 확보.

## 3. 기술적 의의
- **Safe State Machine**: 예약의 상태와 요청의 상태를 분리 관리하고, 재무 정보(`oldTotalPrice`)를 보존함으로써 분산 환경에서도 안정적인 정산이 가능하도록 설계했습니다.
- **Improved UX**: Mustache의 한계를 극복하기 위해 DTO에 불리언 플래그(`isPartialRefund` 등)를 추가하여 사용자에게 꼭 필요한 시점에만 버튼이 노출되도록 제어했습니다.
