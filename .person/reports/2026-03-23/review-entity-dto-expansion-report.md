## 📝 작업 요약
관리자가 AI 분석 결과를 바탕으로 리뷰를 효율적으로 관리하고, 부적절한 리뷰를 삭제(논리 삭제)할 수 있도록 데이터 모델과 전송 객체를 확장했습니다.

## 🛠️ 변경 사항

### 1. Review 엔티티 (`Review.java`)
- **필드 추가**:
    - `aiDangerScore`: AI가 판별한 비방 위험도 (1~5점).
    - `isReviewed`: 관리자가 해당 리뷰에 대해 [삭제/유지] 결정을 내렸는지 여부.
    - `isDeleted`: 논리 삭제 상태 필드.
    - `adminReason`: 관리자가 리뷰를 삭제할 때 입력하는 사유.
- **메서드 추가**:
    - `processByAdmin(isDeleted, adminReason)`: 관리자의 결정을 엔티티에 반영하는 비즈니스 로직.
    - `updateAiScore(score)`: AI 분석 결과를 업데이트하는 로직.

### 2. ReviewResponse DTO (`ReviewResponse.java`)
- **AdminListDTO 추가**:
    - 관리자 전용 리뷰 목록 페이지에서 필요한 데이터를 포함합니다.
    - 작성자 이름, 구역/사이트 정보, AI 위험도, 검토 상태 등을 한눈에 확인할 수 있도록 설계했습니다.

## ✅ 검증 결과
- 엔티티 필드 추가 및 기본값(`@Builder.Default`) 설정 확인.
- DTO 내에서 엔티티 데이터를 UI용 포맷(날짜 등)으로 올바르게 변환하는지 확인.

## 💡 비유로 설명하기
이번 작업은 **'리뷰 게시판'이라는 서류함에 새로운 칸과 도장을 추가한 것**과 같습니다.
- **AI 위험도 칸**: 이 서류(리뷰)가 얼마나 위험한지 AI가 미리 적어두는 칸입니다.
- **검토 도장**: 관리자가 이 서류를 읽고 "통과" 혹은 "삭제" 도장을 찍었는지 표시합니다.
- **사유 메모**: 삭제 도장을 찍을 때 "왜 삭제했는지" 사유를 적어두는 포스트잇을 붙인 것입니다.
이 칸들이 마련되었으니, 이제 AI가 점수를 매기고 관리자가 도장을 찍는 실제 동작을 구현할 차례입니다.
