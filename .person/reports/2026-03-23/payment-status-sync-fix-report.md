# 📄 보고서: 결제 상태 실시간 동기화 및 로컬 웹훅 미수신 해결

**Reporter:** KimNaKim  
**Date:** 2026-03-23  
**Status:** COMPLETED

---

## 1. 작업 요약 (Summary)
실제 결제 테스트 시 `reservation_tb`의 상태가 `PENDING`에서 `CONFIRMED`로 즉시 업데이트되지 않던 문제를 해결했습니다. 특히 로컬 개발 환경(`localhost`)에서 포트원의 웹훅(Webhook) 신호가 서버에 도달하지 못하는 한계를 극복하기 위해, 결제 완료 페이지 진입 시 서버가 직접 포트원 API를 조회하여 상태를 동기화하는 로직을 추가했습니다.

---

## 2. 주요 변경 사항 (Key Changes)

### 2.1 결제 상태 실시간 동기화 메서드 추가 (`PaymentService.java`)
- **`syncPaymentStatus(String impUid)`**: 포트원 결제 고유 번호(`imp_uid`)를 인자로 받아 포트원 API에서 상세 정보를 직접 조회합니다.
- **상태 업데이트**: 해당 예약이 `PENDING` 상태일 경우, 조회된 결제 정보가 '결제 완료(paid)'라면 즉시 `CONFIRMED`로 변경하고 결제 이력을 DB에 기록합니다.
- **중복 방지**: 이미 `CONFIRMED` 이상인 상태에서는 중복 처리를 방지하도록 설계했습니다.

### 2.2 예약 완료 페이지 로직 강화 (`ReservationController.java`)
- **컨트롤러 수정**: `/reservations/complete` 경로로 진입할 때 `paymentId`(`imp_uid`) 파라미터가 포함되어 있으면 즉시 `PaymentService.syncPaymentStatus()`를 호출합니다.
- **효과**: 웹훅 신호가 늦게 오거나(지연), 아예 오지 않는 환경(로컬 개발)에서도 사용자가 완료 페이지를 보는 즉시 예약이 확정됩니다.

---

## 3. 검증 결과 (Validation)
- **로컬 테스트**: 로컬 환경에서 결제 테스트 시, 웹훅이 오지 않더라도 결제 완료 후 `/reservations/complete` 페이지로 리다이렉트되는 즉시 DB의 `reservation_tb.status`가 `CONFIRMED`로 바뀌는 것을 확인했습니다.
- **데이터 정합성**: 포트원의 `customData`를 활용하여 삭제된 예약(선점 만료)에 대한 자동 복구 로직도 동기화 메서드 내에서 동일하게 작동하도록 통합했습니다.

---

## 4. 비유를 통한 기술 설명 (Analogy)
이번 작업은 마치 **"배달원이 도착했다는 벨 소리(웹훅)를 기다리지 않고, 손님이 문을 여는 순간 현관 앞의 택배(포트원 상태)를 직접 확인하는 것"**과 같습니다.

- **기존 방식 (웹훅)**: 배달원(포트원)이 "배달 완료!"라고 벨을 눌러주길 기다리는 방식입니다. 하지만 우리 집(로컬 환경)이 지도에 등록되지 않아 배달원이 벨을 누를 수 없어서 상태가 업데이트되지 않았습니다.
- **개선된 방식 (직접 조회)**: 벨 소리가 들리지 않더라도, 우리가 문을 열고 나갈 때(완료 페이지 진입) 현관 앞에 물건이 있는지 직접 확인하고 "아, 왔구나!"라고 기록(상태 변경)하는 절차를 추가한 것입니다. 이를 통해 벨 소리가 들리지 않는 집에서도 정확하게 배송 완료 처리를 할 수 있게 되었습니다.
