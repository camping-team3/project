# 📄 보고서: 결제 상태 실시간 동기화 및 데이터 파싱 오류(ClassCastException) 해결

**Reporter:** KimNaKim  
**Date:** 2026-03-23  
**Status:** COMPLETED

---

## 1. 작업 요약 (Summary)
실제 결제 테스트 시 예약 상태가 `CONFIRMED`로 업데이트되지 않던 두 가지 핵심 문제를 해결했습니다. 로컬 환경의 웹훅 미수신 문제와 포트원 V2 API 응답 데이터의 형변환 오류를 수정하여 결제 즉시 예약이 확정되도록 보장했습니다.

---

## 2. 주요 변경 사항 (Key Changes)

### 2.1 결제 상태 실시간 동기화 (`PaymentService` & `ReservationController`)
- **수동 동기화**: 결제 완료 페이지 진입 시 서버가 직접 포트원 API를 호출하여 결제 상태를 확인하는 `syncPaymentStatus` 로직을 추가했습니다.
- **로컬 대응**: 이로써 웹훅이 작동하지 않는 로컬 개발 환경에서도 실제 결제 완료 시 예약 상태가 즉시 업데이트됩니다.

### 2.2 포트원 V2 데이터 파싱 오류 해결 (`PortOneServiceImpl`)
- **JSON 파싱 보강**: 포트원 서버로부터 `customData`가 Map 객체가 아닌 JSON 문자열(String)로 전달될 때 발생하던 `ClassCastException`을 해결했습니다.
- **Jackson 활용**: `ObjectMapper`를 사용하여 문자열 형태의 `customData`를 안전하게 Map으로 역직렬화하도록 수정했습니다.
- **금액 추출 최적화**: 포트원 V2의 다양한 금액 필드(`total`, `paid`)를 유연하게 탐색하여 정확한 결제 금액을 추출하도록 보강했습니다.

---

## 3. 검증 결과 (Validation)
- **로컬 테스트**: 실제 토스페이(테스트 결제) 연동 결과, 결제 완료 후 `/reservations/complete` 페이지로 리다이렉트되는 즉시 DB 상태가 `CONFIRMED`로 변경됨을 확인했습니다.
- **로그 확인**: `200 OK` 응답과 함께 포트원으로부터 받은 전체 데이터를 정상적으로 파싱하고 비즈니스 로직에 반영함을 확인했습니다.

---

## 4. 비유를 통한 기술 설명 (Analogy)
이번 작업은 **"외국어로 된 배송 정보를 정확히 번역하여 장부에 기록한 것"**과 같습니다.

- **기존 문제**: 배달원(포트원)이 건네준 배송지 정보(`customData`)가 우리가 읽을 수 없는 암호(JSON 문자열)로 되어 있어서, 이를 우리 장부(Map 객체)에 억지로 끼워 넣으려다 실패했던 상황입니다.
- **해결 방식**: 배송지 정보를 정확히 해독할 수 있는 번역기(`ObjectMapper`)를 도입하여, 어떤 형식으로 정보가 오더라도 정확히 읽어내어 예약 확정 도장을 찍을 수 있게 되었습니다.
