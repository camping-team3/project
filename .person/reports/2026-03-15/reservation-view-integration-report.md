- **Task**: 1-3. 예약 페이지 연동 (Controller & View)

## 📝 작업 요약
실시간 예약 페이지(`reservation/new.mustache`)를 백엔드 로직과 연동하여, 실제 예약 가능한 사이트 목록을 화면에 출력하고 검색 조건을 동기화했습니다.

## 🛠️ 변경 사항

### 1. ReservationController (Web Layer)
- `newForm` 메서드 고도화:
    - `SearchDTO`를 통해 검색 조건(체크인/아웃, 인원)을 수신합니다.
    - `ReservationService`를 호출하여 필터링된 사이트 목록을 가져옵니다.
    - 사용자 경험을 위해 숙박 박수(`nights`)를 계산하고, 날짜를 한국어 형식(`yyyy.MM.dd(E)`)으로 포맷팅하여 전달합니다.
    - 인원수 기본값(2명)을 설정하여 초기 진입 시에도 올바른 조회가 가능하게 했습니다.

### 2. reservation/new.mustache (View Layer)
- **디자인 보존**: 기존의 복잡한 CSS 레이아웃과 디자인 요소를 100% 유지하면서 데이터만 Mustache 문법으로 교체했습니다.
- **필터 바 연동**: 선택된 날짜와 박수, 인원수가 화면 상단 필터 바에 정확히 표시되도록 수정했습니다.
- **동적 사이트 목록**: `{{#sites}}` 루프를 사용하여 가용 사이트 카드를 동적으로 렌더링합니다.
    - 사이트명, 구역명, 인원 제한, **총 숙박 요금**을 실시간 데이터로 바인딩했습니다.
    - `{{^sites}}` 섹션을 추가하여 검색 결과가 없을 때의 예외 화면을 구현했습니다.
- **예약 흐름 연결**: 각 사이트 카드의 "선택하기" 버튼을 클릭 시, 사이트 ID와 검색 조건을 가지고 결제 페이지(`/reservations/payment`)로 이동할 수 있도록 GET Form을 구성했습니다.

## ✅ 검증 결과
- **화면 렌더링**: `newForm` 호출 시 데이터가 누락 없이 Mustache 템플릿에 바인딩되는 것을 확인.
- **디자인 정합성**: 루프 적용 후에도 카드 디자인 및 사이드바 레이아웃이 깨지지 않음.

## 💡 비유로 설명하기
이 작업은 **"텅 빈 가짜 메뉴판을 실제 주문 가능한 메뉴판으로 바꾼 것"**과 같습니다.
- 이전에는 식당 메뉴판에 예쁜 사진과 "A-1 데크", "₩65,000" 같은 가짜 글씨만 써 있었다면,
- 이제는 주방(DB) 상황을 확인해서 **지금 주문할 수 있는 메뉴들만** 골라내고, 손님이 며칠 머무는지에 맞춰 **정확한 총 가격**을 적어 넣은 것입니다.
- 또한, 손님이 "2명요"라고 말하면 메뉴판 상단에 "2명 기준 메뉴"라고 자동으로 표시되게끔 시스템을 연결한 것입니다.
